# Prompt 模板（中文注释版）

> 说明：按场景选择模板。若无特殊要求，优先用模板 A（原生 HTML 单文件）。

## 模板 A：原生 HTML 报告页（默认推荐）

```text
你是资深前端工程师。按以下约束生成可运行的报告页。

[目标]
为{受众}生成{业务主题}的HTML报告，用于{决策目的}。

[页面结构]
{模块1}; {模块2}; {模块3}
布局顺序：{顺序}

[数据接口]
1) {METHOD} {URL}
   params: {参数}
   response: {关键字段}
2) {METHOD} {URL}
   params: {参数}
   response: {关键字段}
字段映射：{UI字段<-API字段}

[交互]
{筛选/时间范围/排序/分页/刷新规则}
状态：loading、empty、error 必须实现。

[技术约束]
- 使用原生HTML+CSS+JS（不使用框架）
- 代码组织：单文件 index.html
- 可读性：语义化结构、关键函数有简短注释

[验收]
- 页面可直接打开运行
- 数据字段完整渲染且格式正确
- 响应式支持 {断点要求}
- 错误与空态可见

[输出协议]
仅输出完整 index.html 代码，不要解释。
```

## 模板 B：React 报告页

```text
你是资深前端工程师。根据下述约束生成 React 报告页。

目标: {目标}
模块: {模块清单}
接口: {接口清单+映射}
交互: {交互规则}
约束:
- React + TypeScript
- 组件拆分: {组件要求}
- 样式方案: {CSS方案}
- 错误/空态/loading 必须覆盖
验收:
- 可运行
- 字段映射正确
- 响应式通过
输出:
1) 先给文件树
2) 再给每个文件完整代码
3) 不要额外解释
```

## 模板 C：两阶段输出（先计划后代码，适合复杂页面）

```text
阶段1（先输出）:
- 用最多10条列出实现计划
- 明确模块、接口调用顺序、状态处理
- 如信息不足，最多提5个关键问题

阶段2（拿到确认后再输出）:
- 生成完整代码
- 严格遵守以下约束: {约束}
- 输出格式: {输出协议}
```

---

## 压缩技巧（用于减少 token）

1. 模块名短语化：`趋势图/明细表/KPI卡`，避免长句解释。  
2. 接口只保留：`method + url + params + response关键字段`。  
3. 验收写成可测试条目，避免形容词。  
4. 输出协议固定一句：`仅输出...不要解释`。  
5. 多条相似规则合并为“统一规则”。
